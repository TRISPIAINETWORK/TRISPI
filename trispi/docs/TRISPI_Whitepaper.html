§ 01

Abstract

TRISPI is a Web4 blockchain network where AI agents act as validators, smart contracts are self-healing, and compute power fuels the intelligence layer. Protected by post-quantum cryptography, TRISPI replaces wasteful Proof of Work with Proof of Intelligence.

Unlike existing Layer-1 networks, TRISPI treats artificial intelligence as a first-class consensus primitive. Every block is validated by a distributed AI ensemble running Proof of Intelligence (PoI), where the quality of machine-learning inference replaces hash computation as the measure of honest participation. Validators earn TRP tokens proportional to the accuracy and consistency of their local AI models.

The network is secured end-to-end with NIST-selected post-quantum algorithms — Dilithium3 for signatures and Kyber1024 for key encapsulation — making TRISPI resilient against quantum adversaries. Smart contracts execute on a dual EVM + WASM runtime, supporting both Solidity and CosmWasm ecosystems.

§ 02

Introduction & Vision

The Web3 era demonstrated that decentralised computation is viable at scale — but the consensus mechanisms that underpinned it consumed enormous energy while contributing no useful work to the world.

2.1 The Problem with Existing Consensus

Bitcoin's Proof of Work burns electricity to solve arbitrary hash puzzles. Ethereum's Proof of Stake requires capital lockup, concentrating power among wealthy validators. Neither mechanism produces useful intelligence — the compute is spent entirely on the consensus game itself.

The AI Opportunity
TRISPI redirects this computational effort toward training and running distributed AI models. Every CPU and GPU cycle spent validating TRISPI transactions also advances the collective machine intelligence of the network.

2.2 Web4 — The Autonomous Layer

TRISPI defines Web4 as the layer where:

  • AI agents autonomously validate transactions and score blocks
  • Smart contracts detect and repair their own vulnerabilities
  • Network topology self-optimises based on real-time trust scoring
  • Users contribute compute and earn proportional rewards in TRP

2.3 Design Principles

§ 04

Proof of Intelligence (PoI)

Proof of Intelligence replaces hash-grinding with local AI inference. Each block is scored by every participating validator using an identical 4-feature weighted model. The consensus score is the trust-weighted median across all validators.

4.1 Scoring Algorithm

The PoI model is a deterministic weighted dot product over four block features:

score = 0.30·tx_quality + 0.25·timing + 0.25·network_health + 0.20·has_proof
FeatureWeightDefinition
tx_quality0.30min(1.0, tx_count / 10) — normalised transaction volume
timing0.251 − |elapsed − block_interval| / block_interval — block arrived on schedule
network_health0.25min(1.0, peers / 10) — peer count health
has_proof0.201.0 if AI proof present, else 0.5

4.2 Multi-Validator Consensus

External energy provider nodes run the same scoring model locally and submit their score to POST /api/validators/submit-score. The server collects all scores for a block and computes a trust-weighted median:

consensus_score = Σ(score_i × trust_i) / Σ(trust_i)

A validator's trust weight (0.10–1.00) updates after each block:

  • Honest score (within 0.25 of median): trust × 1.02 (cap 1.00)
  • Dishonest score: trust × 0.95 (floor 0.10)
  • Trust below 0.30: validator excluded from consensus & earns no TRP

4.3 Validator Rewards

Validators earning TRP per accepted block score:

reward = 0.10 × trust_weight TRP / block

Rewards are posted as signed transactions to the Go blockchain, verifiable on-chain by any node. Server-internal validators ("server_validator") track rewards internally only.

Open Participation
Anyone can run trispi_energy_provider.py to join as a PoI validator. The script generates a fresh Ed25519 keypair, registers with the network, and begins scoring blocks autonomously. No staking requirement (MIN_STAKE_TRP = 0).
§ 05

Post-Quantum Cryptography

All cryptographic primitives in TRISPI are quantum-resistant. Classical elliptic-curve algorithms are augmented — never replaced alone — with NIST-standardised post-quantum schemes.

5.1 Signature Scheme — Dilithium3

Every transaction requires a hybrid signature: a classical Ed25519 signature for backwards compatibility with existing tooling, plus a Dilithium3 (CRYSTALS-Dilithium, NIST Level 3) lattice-based signature for quantum resistance.

5.2 Key Encapsulation — Kyber1024

Node-to-node encrypted channels use Kyber1024 (CRYSTALS-Kyber, NIST Level 5) for key encapsulation. Each Go consensus node establishes a Kyber session key with the Python service on startup:

5.3 Gradient Encryption — AES-256-GCM

Federated Learning gradients are encrypted before submission. Each provider derives a per-round AES-256-GCM key from its public key and the round ID:

round_nonce = sha3_256(f"{round_id}|{provider_id}")
aes_key = sha3_256(pubkey_bytes || round_nonce)

5.4 Address Derivation

TRISPI addresses are derived from the sha3-256 hash of the Ed25519 public key:

address = "trp1" + sha3_256(pubkey)[0:20].hex()

The 40-character hex suffix (160-bit) provides the same address space as Ethereum, ensuring compatibility with EVM tooling while maintaining the trp1 prefix for human readability.

§ 07

EVM + WASM Dual Runtime

TRISPI's Rust Core Bridge provides a unified execution environment for both Solidity (EVM) and CosmWasm (WASM) smart contracts. Developers can deploy existing Ethereum contracts without modification.

7.1 EVM Compatibility

The EVM interpreter in the Rust core is compatible with the Ethereum Yellow Paper, supporting the full EIP-1559 transaction model. Existing Solidity contracts compile and deploy directly. MetaMask connects using Chain ID 7878.

7.2 WASM Runtime

CosmWasm contracts execute in a sandboxed WASM environment. The Rust runtime enforces gas metering, memory limits, and deterministic execution across all validator nodes.

7.3 AI-Augmented Contract Security

The Python AI service continuously monitors deployed contracts with an AI auditor, flagging:

  • Reentrancy vulnerabilities (pattern matching + semantic analysis)
  • Integer overflow / underflow risks
  • Unreachable selfdestruct paths
  • Unusual gas consumption patterns (anomaly detection)
Self-Healing Contracts
When the AI auditor detects a critical vulnerability in a contract, it can trigger an on-chain governance proposal to pause or upgrade the contract automatically — without requiring manual intervention from the developer.

7.4 Hybrid Bridge

The Rust Core Bridge arbitrates cross-runtime calls, allowing Solidity contracts to call CosmWasm modules and vice versa. State is stored in a unified Merkle Patricia Trie shared between both runtimes.

9.2 Peer Scoring

Every peer is assigned a trust score by the AI network intelligence module, based on:

  • Block propagation latency (fast peers score higher)
  • Transaction validity rate (peers relaying invalid txs score lower)
  • PoI score accuracy (consistent vs. outlier scorers)
  • Uptime and responsiveness

Peers whose trust score falls below 0.30 are deprioritised for block relay and eventually disconnected. This Sybil-resistance mechanism prevents dishonest nodes from influencing consensus.

§ 10

Security Model

TRISPI's security model is defence-in-depth: cryptographic guarantees at the transaction layer, AI-driven anomaly detection at the application layer, and trust-weighted consensus at the network layer.

10.1 Byzantine Fault Tolerance

The Go consensus node implements PBFT (Practical Byzantine Fault Tolerance), tolerating up to f malicious nodes in a network of 3f + 1 total nodes. Combined with PoI trust scoring, it is economically irrational for validators to submit dishonest scores — their trust weight decays rapidly (×0.95 per dishonest score) until they are excluded.

10.2 Anti-Poisoning in Federated Learning

The FL aggregation layer employs Byzantine-robust aggregation rules:

  • Gradient norm clipping — gradients with L2 norm > threshold are rejected
  • Cosine similarity filter — gradients diverging significantly from the server's own gradient are flagged
  • Reputation-weighted aggregation — FL reputation score weights each gradient's influence in FedAvg
  • On-chain commitment — aggregation hash committed on-chain ensures aggregation cannot be silently altered

10.3 State Integrity

The blockchain state is protected by a Merkle Patricia Trie (MPT) rooted at each block's state_root. The Python service derives the state root as:

state_root = sha3_256(prev_state_root_bytes || sorted_tx_bytes)

An AI state-integrity guard runs every 30 seconds, verifying the MPT root against the PostgreSQL account balances. Tampering with the account state causes an immediate integrity alert and state rollback.

10.4 Quantum Threat Model

Harvest Now, Decrypt Later
A quantum adversary today could record all encrypted traffic and decrypt it retroactively once a sufficiently powerful quantum computer exists. TRISPI's Kyber1024 KEM provides forward secrecy at post-quantum security levels, neutralising this attack vector.

Shor's algorithm can break Ed25519 on a sufficiently large quantum computer. The Dilithium3 hybrid signature ensures that even if Ed25519 is broken, every transaction and block remains cryptographically valid through the lattice-based signature.
